# Group Policy Management — LGPO Integration

Implementation specification for adding policy-as-code management to libps1
via Microsoft's LGPO utility. Companion to the repository-wide architecture
document.

## Why LGPO and not direct registry writes

libps1 already writes registry values directly via `Set-RegistryValue`. That
works correctly for all policy paths (`Software\Policies\...`) at the
*application* level — Chrome, Firefox, Office, Windows, etc. all read the
same registry locations regardless of who wrote them.

The reason to layer LGPO on top is **auditability**, specifically for the
path toward ISO 27001 certification. Settings written by LGPO appear in:

- `gpresult /h <file.html>` — the supported Windows-native report
- `rsop.msc` — the supported Windows-native GUI
- `Get-GPOReport` (when running against a backup)

Settings written by direct `Set-ItemProperty` do **not** appear in any of
these. They take effect identically, but to an auditor "I wrote it to the
registry directly" requires explaining and trusting a custom script. "I
applied it via LGPO and `gpresult` shows it" is first-class evidence.

For the foreseeable future, both mechanisms coexist:

- **LGPO-managed** — anything under `Software\Policies\...` or
  `Software\Microsoft\Windows\CurrentVersion\Policies\...`, or otherwise a
  documented GP policy. Telemetry, advertising, Defender, Copilot, Edge,
  Chrome, Office.
- **Direct registry (`Set-RegistryValue`)** — Explorer UI tweaks, taskbar
  alignment, Terminal delegation, GameDVR per-user toggles, anything that
  has no documented GP equivalent.

**Hard rule:** anything that has a documented policy equivalent goes via
LGPO, even if direct registry would work. The audit story requires
consistency.

## Directory layout — source vs artifacts

The repository stores **policy sources** as plaintext `.txt` files under
`resources/`. From these, the build pipeline produces **artifacts** for
different delivery mechanisms (local apply, domain GPO backup), written to
the top-level `dist/` directory. The same source produces both.

```
resources/
  policies/                        # AUTHORITATIVE — hand-authored, committed
    00-example.txt                 # example policy source
    01-telemetry.txt               # (planned) telemetry restraint policies
    ...
  metadata/                        # ISO control mapping, justifications, owner
    00-example.psd1                # example metadata file
    01-telemetry.psd1              # (planned)
    ...
  README.md                        # this file — layout, workflow, audit

dist/                              # GENERATED — gitignored wholesale
  GP/
    local/                         # compiled Registry.pol for direct LGPO apply
    gpo-backups/                   # GPO backup folders for domain Import-GPO
```

`resources/policies/` and `resources/metadata/` are committed to git,
reviewed, and audited. Everything under `dist/` is build output —
regenerated by CI on every push, published to a releases archive on tag.
Because `dist/` is a single top-level directory containing only generated
content, `.gitignore` is a single line (`/dist/`) with no risk of
accidentally excluding source.

### Why `.txt` instead of `.lgpo`

Microsoft itself ships policy text sources as `.txt` files alongside the
LGPO tool. The `.lgpo` extension has no formal definition and is not
required by LGPO.exe — it reads any text file via the `/t` flag.
Using `.txt` keeps the format self-documenting, ensures broad editor
compatibility, and avoids confusion with a proprietary extension.

## Policy source file format (`.txt`)

The native LGPO input format. Plaintext, diff-friendly, supports comments
via `;`. Each policy entry is exactly four non-empty, non-comment lines:

```text
; Header — version and ISO control mapping
; Version: 2026-06-14-01
; Justification: Reduces telemetry attack surface; supports ISO A.8.16

; Disable Windows Diagnostic Data collection
Computer
Software\Policies\Microsoft\Windows\DataCollection
AllowTelemetry
DWORD:0

; Disable advertising ID (per-user preference)
User
Software\Microsoft\Windows\CurrentVersion\AdvertisingInfo
Enabled
DWORD:0

; Delete a value that may have been set previously
Computer
Software\Microsoft\Windows\CurrentVersion\Run
SomeUnwantedAutostart
DELETE
```

The four lines per entry are:

1. `Computer` or `User` — which hive scope the value lives in.
2. Registry subkey path (no hive prefix — the scope above determines that).
3. Value name.
4. Value type and data: `DWORD:n`, `SZ:"string"`, `EXSZ:"expand-string"`,
   `MULTISZ:"a","b","c"`, `QWORD:n`, or `DELETE` / `DELETEALLVALUES` /
   `CLEAR`.

### Recommended header conventions

Every `.txt` policy source should include a comment header:

```text
; Version: YYYY-MM-DD-NN
; Title: Short human-readable title
; Owner: <role or name>
; Justification: Why these policies exist. Links to ISO controls.
; ADMX: <FileName>.admx, policy "<PolicyName>"
; Upstream: <URL to Microsoft documentation>
```

## Metadata file format (`.psd1`)

Sidecar metadata for each policy source. Auditor-facing documentation that
doesn't fit cleanly in the `.txt` header. Example:

```powershell
@{
    SourceFile         = '00-example.txt'
    Title              = 'Telemetry and diagnostic data restraint'
    Owner              = '<your name or role>'
    IntroducedVersion  = '2026-06-14-01'
    LastReviewed       = '2026-06-14'
    IsoControls        = @(
        'A.5.34 — Privacy and protection of PII'
        'A.8.16 — Monitoring activities'
    )
    Justification      = @'
...multi-line justification text...
'@
    UpstreamReferences = @(
        'https://learn.microsoft.com/...'
        'ADMX: DataCollection.admx, policy "AllowTelemetry"'
    )
    AppliesTo          = @{
        WindowsEditions = @('Pro', 'Enterprise', 'Education')
        MinBuild        = 19041
    }
}
```

## Domain compatibility

The policy text files are **value-level** domain compatible — the registry
paths (`Software\Policies\...`) are identical whether applied locally or by
a domain GPO. The *delivery* differs:

- **Local apply:** `LGPO.exe /t source.txt` writes directly to local
  `Registry.pol`.
- **Domain apply:** the same source is compiled to a `Registry.pol`,
  wrapped in a GPO backup folder structure (`Backup.xml`, `bkupInfo.xml`,
  `gpreport.xml`, `DomainSysvol/GPO/Machine|User/registry.pol`), and
  imported into AD via `Import-GPO -BackupGpoName ...`.

### Domain build pipeline — `Build-GroupPolicyBackup.ps1`

The domain apply side is handled by `scripts/Build-GroupPolicyBackup.ps1`,
a work-in-progress "compiler" that converts policy text sources into
domain-importable GPO backup folders.

At a high level, the pipeline works as follows:

1. **Skeleton copy.** A vendored reference GPO backup skeleton (created once
   in GPMC from an empty GPO, then committed under
   `resources/policies/skeleton/`) provides valid `Backup.xml`,
   `bkupInfo.xml`, and `gpreport.xml` stubs. The build script copies this
   skeleton into a staging area and assigns a fresh GUID.

2. **Registry.pol generation.** LGPO.exe compiles the policy text sources
   (`.txt` files under `resources/policies/`) into `Machine\registry.pol`
   and `User\registry.pol` binaries, which are placed into the copied
   skeleton's directory structure.

3. **Metadata rewrite.** The skeleton's XML metadata is rewritten with the
   correct display name, GUID, and timestamp for this build.

4. **Output.** The completed GPO backup folder is placed in
   `dist/GP/gpo-backups/` (top-level, gitignored). From there, a per-company
   AD deployment repo picks it up and runs `Import-GPO`, `New-GPLink`, and
   security filtering.

The script is currently stubbed — both TODO(1) (skeleton copy) and TODO(2)
(LGPO compilation + metadata rewrite) throw intentionally. Implementation
is planned as the next phase after the local-apply LGPO functions are
validated end-to-end.

The local apply is implemented now via `Invoke-LGPO`; domain build is in
progress via `Build-GroupPolicyBackup.ps1`.

### Platform notes

- **Domain-only policies** (account lockout, Kerberos, password
  complexity): set as separate text files when authored, omitted from the
  local apply set, included only in the domain build.
- **GPO-level features** (security filtering, WMI filtering, loopback
  processing, link order): these are properties of the GPO object in AD,
  not of the registry values inside it. They're declared in the build step
  that wraps a source into a GPO backup, or applied at link time by the
  domain admin.

## Policy authoring workflow

1. Author or modify a `.txt` file in `resources/policies/`. Include the
   version comment in the header.
2. Author or update the matching `resources/metadata/*.psd1` with the ISO
   control mapping.
3. Test in a VM via `Invoke-LGPO -PolicyPath <file>`.
4. Verify the apply with `gpresult /h gpresult.html` and inspect the
   relevant section. The values from your policy source should appear in
   the report.
5. Commit both files in the same change. Reviewer checks both.

## LGPO behavior notes

### Additive application

`LGPO.exe /t <file.txt>` **adds or updates** the values in the file. It
does **not** remove values that a previous apply set but that are absent
from the current file. So if you remove a policy entry from a source and
re-apply, the old value stays in `Registry.pol` — the source no longer
reflects the actual applied state, which is an audit problem.

Two strategies for authoritative apply:

- **Wipe-then-apply:** before applying, delete the local policy store,
  then apply the full source set:
  ```powershell
  Remove-Item "$env:SystemRoot\System32\GroupPolicy\Machine\Registry.pol" -Force
  Remove-Item "$env:SystemRoot\System32\GroupPolicy\User\Registry.pol"    -Force
  Invoke-LGPO -PolicyPath <file.txt>
  gpupdate /force
  ```
  Clean and predictable. Suitable for image builds and maintenance windows.

- **Explicit DELETE entries:** when retiring a policy, change its block to
  a `DELETE` directive so the next apply actively removes the value. More
  surgical, preserves audit continuity (the history shows the value was
  explicitly retired), but requires discipline.

For the initial migration, **additive application is acceptable** because
you're adding policies, not retiring them. Adopt the wipe-then-apply
pattern (or DELETE-directive discipline) once the policy set is mature.

### LGPO `/t` flag

The `/t` flag tells LGPO.exe to apply a text-format policy source (our
`.txt` files). The `/g` flag applies a GPO backup directory. `Invoke-LGPO`
auto-detects which to use based on whether `PolicyPath` is a file or
directory.

## Migration sequencing

Concrete order to move existing direct-registry policies into LGPO. This
is months of work in total — but the function plumbing is done, and each
individual migration is a small, reviewable change.

1. Author the first policy baseline — pick something small like
   advertising-info disable. Run it through `Invoke-LGPO`. Verify via
   `gpresult /h`. This validates the end-to-end pattern.
2. Migrate one category at a time from direct-registry to LGPO. Suggested
   order:
   1. Telemetry (largest blast radius for the audit story)
   2. Defender
   3. Edge / Chrome / Firefox
   4. Copilot and Edge Copilot
   5. Office
   6. Advertising / suggestions / Cloud Content
3. Leave non-policy items on direct registry — Terminal delegation,
   Explorer UI tweaks, GameDVR per-user toggles. These have no GP
   equivalent.

## Audit concerns

### Versioning the LGPO binary

The weekly CI (via `Test-LGPOSourceAvailability`) catches URL availability.
The content-drift hash check ensures the downloaded binary matches the
committed hash. A failed match means Microsoft updated the binary —
pause, review, update `Resolve-LGPOSource`'s hash table, commit.

### Versioning the policy baselines

Each `.txt` file carries a `; Version: YYYY-MM-DD-NN` comment.
`Invoke-LGPO` logs which file (and therefore which version) was applied.
Combined with the structured return object's timestamp, you can answer
"what policy was in effect on date X?" from logs alone.

### Intent vs observation

The `.txt` files in git are the *intent*. `gpresult` output post-apply is
the *observation*. Proving they match is the *control*. The planned
(audit-report generator) compares parsed `.txt` against parsed
`Get-LGPOAppliedState` output to produce a configured-vs-observed diff.

### ADMX mapping

Auditors often want to see the link between your policy values and
Microsoft's published ADMX templates. Include the ADMX file name and
policy name in the `.txt` comment header (and again in the metadata
`psd1`):

```text
; ADMX: DataCollection.admx
; Policy: AllowTelemetry
; Microsoft documentation: https://learn.microsoft.com/...
```

### Don't auto-update LGPO in production

Pin the version. The CI tells you when an update is available; a human
reviews, runs tests, and commits the version bump. "We track upstream
dependencies but apply updates only after review" is a stronger control
claim than "we always use the latest".
